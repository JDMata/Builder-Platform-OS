## Constitutional documents

**Established 2026-07-17**, before Sprint 2 planning begins; **frozen as v1.0 and patched to v1.0.1 on 2026-07-17** following a Constitutional Review (governance/terminology/versioning fixes only, no conceptual redesign — see `ENGINEERING_DECISION_LOG.md` ED-014). This is the canonical, authoritative list of the constitutional set — individual constitutional documents deliberately do not restate this count or enumeration inline, to avoid drifting out of sync with it as the set grows. Four documents currently govern the platform at the same authority level, none overridable without the same deliberate, recorded governance process used to change any one of them, and each authoritative only within its own declared domain (see each document's own Governance section for its constitutional-precedence and conflict-resolution rule):

- [docs/architecture/00-vision-and-principles.md](docs/architecture/00-vision-and-principles.md) — the Architecture Foundation (AF v1.0.1): the engineering domain — architecture, code structure, data/domain modeling, service boundaries, infrastructure.
- [docs/ux/00-platform-experience-foundation.md](docs/ux/00-platform-experience-foundation.md) — the Platform Experience Foundation (PXF v1.0.1): the experience-philosophy domain — vision, personas, information architecture, AI interaction model, design/interaction/accessibility/motion principles, design tokens strategy, governance, and a Definition of Done for UX, governing every screen built from Sprint 2 onward.
- [docs/product/00-engineering-canvas-specification.md](docs/product/00-engineering-canvas-specification.md) — the Engineering Canvas Specification (ECS v1.0.1): the Engineering Canvas's work-model domain — the product specification for the platform's living engineering workspace, the human experience layer over the Project Digital Twin ([ADR-0021](docs/adr/0021-project-digital-twin-knowledge-graph.md)) — its graph model, node/relationship taxonomy, the four synchronized views (Guided, Canvas view, Documents, Executive), AI annotation behavior, keyboard-first interaction, inspector panels, timeline model, extensibility, and a Definition of Done.
- [docs/ux/01-experience-language-specification.md](docs/ux/01-experience-language-specification.md) — the Experience Language Specification (XLS v1.0.1): the visual-language domain — the platform's visual, interaction, motion, and communication language, typography, color and semantic color, material/glass/elevation, motion grammar, iconography/illustration, AI visual language, the Canvas view's visual expression, and Figma/React derivation rules — underneath both the PXF's general screens and the ECS's Canvas.

None of the four produces screen designs, component code, or implementation detail — all four are principle-first and technology-agnostic; downstream Figma/React work must be derivable from them, not an independent design exercise. Every future Vertical Slice's Product Design Review checks against all four.

## Product Architecture Specification (derivative, not constitutional)

[docs/product/01-product-architecture-specification.md](docs/product/01-product-architecture-specification.md) — the PAS (v1.0): the first artifact derived *from* the four constitutional documents rather than governing alongside them. It translates the Constitution into a complete product architecture (Lifecycle Workspaces, object model, workspace contracts, capability model, navigation model, an end-to-end idea-to-production walkthrough, AI architecture, and product governance), and in doing so validates the Constitution's sufficiency. It found the constitutional layer's *product model* complete but identified five concrete gaps in its *capability catalog* (no named generation capability for Architecture Decisions, Technical Specifications, Development-phase artifacts, or Test suites; no Development/Testing/Deployment personas; an incomplete event catalog; no Continuous Improvement/Lessons Learned grounding) — see the PAS's own closing verdict. Unlike AF/PXF/ECS/XLS, the PAS is revised freely as these gaps close and carries no constitutional authority of its own.

**Operational catalogs (implementation-facing, derived from the PAS):** every `[GAP]` the PAS identified was re-examined and reclassified — none required inventing a new constitutional principle, since the Constitution already defines where each would live and how it would be added (a capability registry entry, a PXF persona-table row, an event-catalog row, or a Platform Pack), except one (Continuous Improvement/Lessons Learned), which remains a genuine `Future Extension` requiring constitutional-weight change. See [02-capability-catalog.md](docs/product/02-capability-catalog.md) (CAP v1.0, 14 capabilities + roadmap), [03-persona-catalog.md](docs/product/03-persona-catalog.md) (PER v1.0, 5 existing + 3 Planned personas), [04-transition-ownership-matrix.md](docs/product/04-transition-ownership-matrix.md) (TOM v1.0, all 6 Lifecycle Workspace transitions fully owned), and [05-operational-readiness-assessment.md](docs/product/05-operational-readiness-assessment.md) (consistency validation, coverage assessment, recommended Sprint 2 backlog: `Generate Functional/Technical Specification`, a documentation/architecture event-catalog addition, and closing `CI-B5`).

**Canonical Machine Model (technology-neutral contract layer):** [06-canonical-machine-model.md](docs/product/06-canonical-machine-model.md) (CMM v1.0) formalizes every entity, relationship, identifier, versioning, and lifecycle rule the constitutional and product-architecture layer already implies — into a form a TypeScript service, a Java service, a Python agent, a graph database, or a Platform Pack author can consume identically, without re-deriving it from prose. Every element cites AF/PXF/ECS/XLS/PAS/CAP/PER/TOM, an ADR, or a real shipped artifact (the event model, the one real `Capability`, the one real `AgentDefinition`); three narrow, explicitly-named gaps remain open (no branching model, no stated deletion policy for `Capability`/`CapabilityProvider`, and JSON Schema as the canonical schema language is a judgment call rather than a direct citation).
