# 0021 — Project Digital Twin as an event-projected knowledge graph
Status: Accepted
Date: 2026-07-14

## Context

Every project must have a Digital Twin representing every artifact produced across its lifecycle (requirements through incidents) with typed relationships between them (`Requirement implements CAP Service`, `Requirement validated-by Test`, `Deployment contains Application Version`, ...). Without a designed approach, the natural failure modes are: duplicating data already owned by existing aggregates (`Requirement`, `Artifact`, `WorkflowRun`) into a second, drifting copy; an unbounded, ad hoc set of relationship-type strings that no two contexts spell the same way; and a graph that becomes untrustworthy because AI-suggested relationships are indistinguishable from human-confirmed ones. Full design: [16-project-digital-twin.md](../architecture/16-project-digital-twin.md).

## Decision

The Digital Twin is a new bounded context — **Digital Twin / Traceability** — that owns the *graph structure* only (`DigitalTwinNode`, `DigitalTwinEdge`, a `NodeTypeDefinition` and `RelationshipTypeDefinition` registry, `DigitalTwinSnapshot`) and never the artifact content itself. A node's real data stays owned by whichever existing context produced it (`Requirement`, `Artifact`/`ArtifactVersion`, `WorkflowRun`, `Deployment`, `Risk`, ...); the graph stores an opaque `sourceRef` plus minimal denormalized display fields (label, status), exactly the read-model-projection relationship [ADR-0014](0014-cqrs-read-models.md) already established between write-side aggregates and reporting projections — graph-shaped instead of tabular-shaped. The graph is populated by subscribing to the same domain events every context already emits ([ADR-0007](0007-event-driven-transactional-outbox.md)), plus a small number of new events for artifact types not yet modeled.

Both node types and relationship types are **opaque, registry-declared strings**, never a hardcoded core enum — the third application of the same pattern already used for `ArtifactType` ([05-plugin-architecture.md](../architecture/05-plugin-architecture.md)) and `PortCategory` ([ADR-0019](0019-execution-profiles-for-generated-applications.md)). This is what keeps SAP-specific taxonomy (`"cap-service"`, `"odata-entity"`) out of the platform's core code while still letting the graph be fully typed and queryable. Every edge carries `provenance: 'declared' | 'ai-inferred'` and, for inferred edges, a confidence score — an inferred relationship is never treated as ground truth for impact analysis or traceability-completeness checks until confirmed via the existing `ReviewGate` mechanism.

Storage: a new `digital_twin` Postgres schema (consistent with schema-per-context, [ADR-0009](0009-postgresql-schema-per-context-drizzle.md)), queried via Apache AGE (an open-source, Cypher-compatible graph extension that runs inside the same Postgres instance) rather than standing up a dedicated graph database as a new operational dependency — behind a new `ports/graph-store.port.ts`, so a dedicated graph engine can be adopted later as an adapter swap if traversal performance at scale demands it, without touching any consumer.

New lightweight aggregates required to cover artifact types with no existing home: `Risk`, `Incident`, `Problem`, `Change` (Governance & Audit context — the concrete, ITIL-aligned realization of a principle already stated but not yet modeled) and `Deployment`, `ApplicationVersion` (Project & Workspace context, alongside the existing `Environment`). Everything else in the user's example list (Business/Functional/Non-Functional Requirements, User Stories, Acceptance Criteria, Architecture, FS, TS, UI Screens, CAP Services, Entities, OData Services, Tests, Documentation, User Manuals, Demo Scripts, Recordings) is represented by **reusing** the existing `Requirement` (with a `kind` field) and `Artifact`/`ArtifactVersion` (with new `artifactType` values — no schema change, since `artifactType` was already designed as an open string) rather than inventing a new aggregate per artifact type.

## Consequences

- Full lifecycle traceability (business requirement → generated code → test → deployment → incident) becomes queryable, not just implied — the concrete mechanism behind the PMO/ITIL-alignment principles stated in [00-vision-and-principles.md](../architecture/00-vision-and-principles.md) since Sprint 0.
- Impact analysis ("what breaks if I change this Entity") and root-cause assistance ("what changed before this Incident") become graph traversals an agent can perform via a `digital-twin-search`/`graph-query` MCP tool, composing directly with the AI Workspace ([ADR-0020](0020-ai-workspace-for-agent-definitions.md)) rather than requiring a new agent-integration mechanism.
- Adds a real new bounded context and, with it, real ongoing discipline: every new node/relationship type must be registered (checked in CI, like `ArtifactType`), and the graph must never become a second place artifact content is edited — only origin contexts write their own data; the graph only ever projects.
- Because nodes reference source data by ID rather than duplicating it, the graph is only as fresh as its event-projection lag (same eventual-consistency caveat already accepted for read-models in [ADR-0014](0014-cqrs-read-models.md)) — acceptable for traceability/impact-analysis use cases, which don't require read-your-writes consistency.

## Alternatives considered

- **Model every artifact type as its own new bounded-context aggregate** (a `UserStoryAggregate`, `UIScreenAggregate`, `CAPServiceAggregate`, ...): rejected — most of the user's example list is a shape or type variant of a concept that already exists (`Requirement`, `Artifact`); inventing a dozen-plus new aggregates for what are really `kind`/`artifactType` values would be the same "new context per new noun" over-fragmentation already corrected once in this project ([ADR-0018](0018-package-granularity-revision.md)).
- **A dedicated graph database (Neo4j, Amazon Neptune) from the start**: rejected for now — no current scale justifies a new operational dependency; Apache AGE gets real graph query semantics (Cypher, variable-length path traversal) without leaving Postgres, and the port abstraction keeps the option open, the same reasoning already applied to the workflow-engine and event-transport decisions.
- **RDF/OWL triple store with formal ontology reasoning**: rejected — heavier than the platform's actual near-term need (traceability and bounded-depth impact analysis, not formal semantic inference); a property graph is the pragmatic default, consistent with "boring technology in the core."
- **Let AI-inferred relationships be written directly as regular edges**: rejected — indistinguishable-from-declared inferred edges would silently corrupt impact analysis and traceability-completeness checks the first time an inference is wrong; the `provenance`/confidence/`ReviewGate` split is required.
