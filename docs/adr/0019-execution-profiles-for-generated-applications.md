# 0019 — Execution profiles for generated applications (Local POC / Hybrid / Enterprise)
Status: Proposed
Date: 2026-07-14

## Scope note (read first)

This ADR concerns the architecture of **applications generated by SAP App Factory's plugins** — the CAP services, Fiori apps, and eventually RAP/ABAP and Integration Suite artifacts that plugins produce as output. It does **not** change the platform's own infrastructure, which already uses PostgreSQL/Redis/MinIO/OIDC per [ADR-0009](0009-postgresql-schema-per-context-drizzle.md) and [ADR-0010](0010-oidc-federation-zero-trust.md) and is unaffected by this decision. Full design: [14-execution-profiles.md](../architecture/14-execution-profiles.md).

## Context

Generated applications must run in at least three materially different environments without forking their business logic:

1. **Local POC** — entirely on a developer's machine: CAP Local, SQLite, mock authentication, mock roles, mock SAP APIs, mock external APIs. The generated app must feel almost identical to its enterprise counterpart, not like a toy.
2. **Hybrid** — running locally, with some services calling real SAP/external systems and others still mocked.
3. **Enterprise** — running on SAP BTP: HANA Cloud, XSUAA/IAS, Destination Service, Connectivity Service, Integration Suite.

Without an explicit architecture for this, plugin authors will each solve it differently (or not at all), generated apps will hardcode SQLite-specific or HANA-specific assumptions into business logic, and "move from SQLite to HANA Cloud" will mean a rewrite rather than a configuration change — precisely the kind of vendor lock-in and technical debt [ENGINEERING_PRINCIPLES.md](../../ENGINEERING_PRINCIPLES.md) exists to prevent, now surfacing in the platform's *output* rather than its own core.

## Decision

Every capability plugin that generates a runnable application (CAP Node/Java, Fiori, and future generators) must scaffold that application internally as **Ports & Adapters**, with exactly seven port categories: **Persistence, Authentication, Authorization, Messaging, Storage, SAP Connectivity, External APIs**. The generated app's domain/business logic depends only on these ports and contains no SAP-specific or infrastructure-specific code — the same rule [ADR-0006](0006-plugin-architecture.md) already applies to the platform's own core, now applied to the platform's output.

An **Execution Profile** is a named, per-port-category adapter selection. Three profiles are provided as presets — `local-poc` (every port bound to a mock/local adapter: SQLite, mocked auth/roles, mocked SAP/external APIs), `enterprise` (every port bound to a real BTP adapter: HANA Cloud, XSUAA/IAS, Destination + Connectivity Service, Integration Suite), and `hybrid` (an arbitrary per-port mix of the two — not a distinct mechanism, just a non-uniform point in the same configuration space). This composability is deliberate: supporting Hybrid requires no additional design once each port independently resolves to an adapter — it falls out of the Local/Enterprise mechanism for free.

Adapter implementations for these seven ports are **not** reimplemented per plugin. A shared, versioned package (`packages/generated-app-kit`, owned by the platform, consumed by generated applications as a runtime dependency) provides the port interfaces and both the mock/local and thin-wrapper-over-official-SAP-SDK adapters. Plugins scaffold business logic that depends on this kit's ports; they do not hand-roll their own mock-auth or mock-persistence implementations. Enterprise-tier adapters for SAP Connectivity resolve real credentials through the existing `TargetSystemConnection` mechanism ([ADR-0015](0015-target-system-credential-management.md)) — this decision does not introduce a second place to store customer-system credentials.

The core platform's domain/application layers know that execution profiles exist (a `PluginManifest` declares which profiles a plugin supports; `GenerationInput` specifies a target profile; `Artifact`/`ArtifactVersion` record which profile they were generated for) but never know what SQLite, HANA, XSUAA, or CAP profiles actually are — that mapping is entirely inside each plugin's implementation and inside `generated-app-kit`, never in `packages/context-*`. Full domain-model and mapping detail: [14-execution-profiles.md](../architecture/14-execution-profiles.md).

## Consequences

- Generated applications gain a real, enforceable path from Local POC to Enterprise: swapping the Persistence port's binding from `sqlite` to `hana-cloud` is a profile-config change, not a business-logic rewrite — this is the concrete mechanism behind "the generated applications must move from SQLite to HANA Cloud with minimal changes."
- Local POC demos are materially more convincing (mocked but structurally identical to production, not a simplified toy version), which directly serves the platform's sales/adoption motion.
- Adds real scaffolding cost: every application-generating plugin must produce seven ports' worth of wiring plus profile configuration, not just business logic — mitigated by `generated-app-kit` doing the heavy lifting once, shared across plugins.
- Introduces a new drift risk — mocked and real adapters for the same port must behave identically at the contract level, or "works in Local POC, fails in Enterprise" becomes a systemic bug class. Addressed by extending the platform's existing contract-test discipline (`testing-kit`) to `generated-app-kit`'s adapters, and discussed further as a self-review risk in [14-execution-profiles.md](../architecture/14-execution-profiles.md).
- Plugins that generate non-runnable artifacts (documentation, architecture diagrams) or artifacts with no realistic Local POC story (e.g., a RAP/ABAP generator, where there is no meaningful "local ABAP system") are not required to support all three profiles — profile support is declared per plugin in its manifest, not assumed universal.

## Alternatives considered

- **Let each plugin invent its own local/mock strategy independently**: rejected — guarantees inconsistent quality across plugins, duplicated mock-adapter implementations, and no shared contract-test discipline to catch mock/real drift; directly contradicts the platform's own no-duplicated-cross-cutting-concern practice already established for `llm-core`/`mcp-core` resilience ([ADR-0016](0016-mandatory-resilience-patterns.md)).
- **Three hardcoded, monolithic profile bundles with no per-port composability**: rejected — cannot express "Hybrid" (some services real, some mocked) without either a fourth+ bundle per combination (combinatorial explosion) or an escape hatch that undermines the whole model. Per-port composability handles Hybrid for free.
- **A second, execution-profile-specific credential store for Enterprise-tier SAP Connectivity**: rejected — [ADR-0015](0015-target-system-credential-management.md) already solves privileged target-system credential storage with envelope encryption and BYOK; a second mechanism would duplicate that security investment and create two places for the same class of secret to live, exactly the kind of debt this platform's governance exists to prevent.
- **Require every plugin to support every execution profile**: rejected — some generator types (RAP/ABAP) have no realistic local-execution story; forcing a fictional "Local POC" for them would produce a mock so unlike the real system it provides no value, undermining the "feels almost identical to enterprise" requirement rather than serving it.
