  **A real, previously-undiscovered cross-package bug, found only because this story actually ran two `persistence-postgres/*` packages' real-Postgres suites back to back in one continuous session — something no single story's own verification had ever done before:** `drizzle-orm`'s `migrate()` defaults to one shared, unscoped tracking table (`"drizzle"."__drizzle_migrations"`) regardless of which package calls it. Running `persistence-postgres/identity`'s `migrate()` first (as the new `integration` job's alphabetical `pnpm --filter` ordering does) left rows in that shared table; `persistence-postgres/governance`'s own `migrate()` call then compared its 2-migration journal's length against that unrelated row count, concluded — wrongly — that it had nothing left to apply, and **silently never created the `governance` schema at all**. No error thrown anywhere: `migrate()` simply resolved having done nothing, and the failure only surfaced downstream as `error: schema "governance" does not exist` in the test file's own `beforeEach`. Root-caused by running `migrate()` standalone via `tsx` outside Vitest's reporting layer (confirming it really did resolve without creating the schema, ruling out a swallowed exception) and by inspecting `drizzle.__drizzle_migrations` directly via `psql`. A first fix attempt (`migrationsSchema` scoped to each context's own schema) was tried, verified, and **rejected**: it collides with these packages' own migration files, which already do their own unconditional `CREATE SCHEMA "identity"`/`"governance"` — drizzle's schema-bootstrap step for the tracking table creates that same schema first, so the migration file's own statement then fails with "schema already exists." The actual fix: keep the shared `drizzle` schema, give each package a distinct `migrationsTable` (`identity_migrations`, `governance_migrations`). Re-verified from a fully clean `infra:reset` → `infra:up`, running identity then governance in sequence: both schemas now created correctly, both suites pass (8/8, 9/9). Documented as the central fact in both packages' READMEs, the same reflex as SAF-14's superuser/RLS finding — this is exactly the kind of bug a CI pipeline running real multi-package integration tests exists to catch, and did.

  **A second real, found tooling gap:** `pnpm audit` (any form, any flag) fails outright with `410 Gone` — verified by hand, independent of anything this story changed. npm retired the legacy per-package audit endpoint `pnpm audit` calls, in favor of a bulk advisory endpoint pnpm doesn't yet speak.
