Every RLS assertion silently passed for the wrong reason until this was caught (by noticing the isolation test failed once the *actual* bug — see below — was fixed, which shouldn't have been possible if RLS were truly being exercised). Fixed by adding a second, non-superuser role (`saf_app`, provisioned via `infra/docker-compose/postgres-init/`) that every repository connects as at runtime, with the migration role reserved for schema changes only. Documented as the central fact in both packages' READMEs — this is the one finding future contributors most need to not silently regress.

  **Found and fixed, all against the real database:** (1) `RequestContext.tenantId` is `string`, never UUID-constrained, but `09-database-proposal.md`'s illustrative schema said `tenant_id uuid` — corrected to `text` (and the RLS policy example corrected to match), consistent with the same `id`-typing lesson from SAF-11's outbox table. (2) The RLS policy's `WITH CHECK` (implicit from `USING` on a `FOR ALL` policy) rejected every insert at first — `TenantRepository.save()` was writing `tenant_id: tenant.id` (the aggregate's own identity) while the session variable was set to `ctx.tenantId` (the calling session's tenant), and the two aren't always the same value; fixed by storing `ctx.tenantId` instead, and documented *why* they're kept as two separate concepts (onboarding a brand-new tenant is inherently a platform-admin operation performed by a session that isn't that tenant yet). (3) `AuditEvent`'s partitioned primary key `(id, created_at)` means `id` alone has no database-enforced uniqueness — `save()` can't `ON CONFLICT (id)` the way `TenantRepository`'s can; documented as a deliberate, known Sprint 0 limitation (real dedup needs a time-ordered id scheme or an explicit step, neither built here) rather than silently glossed over.

  Verified: full monorepo `build`/`typecheck`/`lint:eslint`/`lint:deps`/`format:check`/`test` all pass (25 workspace projects; dependency-cruiser required no rule changes — `persistence-postgres` was already in `ADAPTER_TIER_PATH`, though `pnpm-workspace.yaml` did need its own one-level-deeper glob, the same SAF-9 lesson applied proactively this time). Both packages' real-Postgres suites re-verified from a completely fresh `pnpm infra:reset` → `up` cycle, proving the migration story holds from scratch, not just against an already-provisioned database. No ADR change — implements ADR-0009/ADR-0013/ADR-0017 as already decided; the superuser/RLS finding and the schema `tenant_id` type correction are real-world corrections to already-decided designs, not new decisions.
