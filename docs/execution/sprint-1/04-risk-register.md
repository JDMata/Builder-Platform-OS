# Sprint 1 Risk Register

**Revised 2026-07-16** for the merged, single-slice task numbering ([02-vertical-slice-catalog.md](02-vertical-slice-catalog.md)). Consolidates risks already on record ([12-risks-and-technical-debt.md](../../architecture/12-risks-and-technical-debt.md), [Known Risks](../../governance/sprint-0-exit-gate/07-known-risks.md), the Product Design Review's UX/Future risk findings) with new risks specific to this execution plan, including two introduced by the Digital Twin/Project Type pull-forward.

| # | Category | Risk | Probability | Impact | Mitigation | Owner |
|---|---|---|---|---|---|---|
| 1 | **Business** | Discovery structures requirements the business stakeholder didn't actually mean, and it isn't caught before confirmation. | Low | High | The `Clarification` mechanism catches ambiguity before it's silently resolved; User Confirmation (Task 1.16) is the human checkpoint; "Request Changes" always available. | Product Owner |
| 2 | **Business** | The real LLM call (Task 1.5) incurs unexpected cost at scale. | Medium | Medium | `CostBudget`/`UsageRecord` mechanism already designed in the LLM Gateway context (Sprint 0); confirm budget guardrails are active before any real traffic. | Backend Engineer (Task 1.5 owner) |
| 3 | **UX** | The Discovery experience reads as "a document generator with an AI assist" rather than "a consulting platform" if Quick Wins are skipped or rushed. | Medium | Medium | Quick Wins are explicit acceptance criteria on Tasks 1.6, 1.12, 1.16, 1.21 — not optional polish. | Frontend Engineer (Task 1.12/1.16/1.21 owners) |
| 4 | **UX** | Accessibility gaps (placeholder-as-label, unlabelled clarification inputs, unstructured confirmation content) ship if not explicitly verified. | Medium | Medium | Each fix is a named acceptance criterion on Tasks 1.10, 1.12, 1.16 — not a general "accessibility pass." | Frontend Engineer |
| 5 | **Architecture** | Real LLM integration (Task 1.5) reveals `LlmProviderPort`'s contract doesn't actually fit a real provider's behavior. | Low | High | **Stop, per [.ai/README.md](../../../.ai/README.md)'s "stop, explain impact, recommend an ADR" rule** — do not patch around a port mismatch silently. | Backend Engineer (Task 1.5 owner) |
| 6 | **Architecture** | `CapabilityResolverPort`'s first real adapter (Task 1.7) reveals the port's contract doesn't generalize past the single-provider case it was designed against. | Low | Medium | Same stop-and-recommend-ADR rule; the new contract-test suite (Task 1.7) is designed to surface this early. | Backend Engineer (Task 1.7 owner) |
| 7 | **Architecture** | The minimal `context-digital-twin` implementation (Tasks 1.17/1.18), built ahead of Sprint 7, drifts from what [ADR-0021](../../adr/0021-project-digital-twin-knowledge-graph.md) actually specifies, because it's being built under schedule pressure rather than as originally planned. | Medium | High | Task 1.17's acceptance criteria explicitly require matching ADR-0021's model exactly, with no node/relationship type beyond this slice's real need — reviewed against the ADR text directly, not from memory. If ADR-0021's design doesn't actually fit once real implementation starts: **stop and recommend an ADR revision**, do not improvise around the gap. | Backend Engineer (Task 1.17/1.18 owner) |
| 8 | **Architecture** | `Project.projectType` (Task 1.13) is implemented as a hardcoded Kernel-level enum instead of an opaque, Platform-Pack-supplied string, silently reintroducing the SAP-specific Kernel leakage [ADR-0023](../../adr/0023-platform-kernel-and-platform-pack-architecture.md) confirmed wasn't present. | Low | Medium | Task 1.13's acceptance criteria explicitly forbid a hardcoded union/enum for `projectType`; enforced at code review, same as any other architectural rule. | Backend Engineer (Task 1.13 owner) + PR reviewer |
| 9 | **Workflow** | The clarification loop (Tasks 1.11/1.12) doesn't terminate for a pathological or adversarial idea input. | Low | Medium | Bound the number of clarification rounds explicitly (application-layer safeguard); surface a clear "needs manual review" outcome past the bound. | Backend Engineer (Task 1.11 owner) |
| 10 | **Workflow** | `WorkflowRun` state gets corrupted by a concurrent clarification answer and a workflow-engine timeout racing each other, now across a longer, single continuous run (intake through Digital Twin update). | Low | Medium | The in-memory workflow engine's existing state-transition guarantees are exercised by a real, longer multi-step run for the first time — verify explicitly in Task 1.20's testing, not assumed to hold. | Backend/Workflow Engineer (Task 1.20 owner) |
| 11 | **Security** | Anthropic API credentials are mishandled during Task 1.5's real-call implementation. | Low | High | Mandatory: credentials only through `SecretsVaultPort`; PR review checklist ([PROJECT_PLAYBOOK.md](../../../PROJECT_PLAYBOOK.md)) checks this specifically. | Backend Engineer (Task 1.5 owner) + PR reviewer |
| 12 | **Security** | The new `CapabilityResolverPort` or `GraphStorePort` adapter resolves/writes without the same Zero Trust `RequestContext` discipline every other port already requires. | Low | Medium | `RequestContext` is a mandatory first parameter on every port method by existing convention — verified structurally at code review. | Backend Engineer (Task 1.7/1.18 owners) |
| 13 | **Documentation** | The four new packages (`persistence-postgres/requirements`, `persistence-postgres/project`, `context-digital-twin`, the `GraphStorePort` adapter) ship without READMEs, failing `require-package-readmes.mjs`. | Low | Low | Already mechanically enforced by CI — cannot merge without it. | Owners of Tasks 1.4, 1.14, 1.17, 1.18 |
| 14 | **Testing** | Real-LLM-backed tests (Tasks 1.5, 1.6) are flaky or slow, discouraging developers from running them. | Medium | Medium | Follow the same env-gated real-provider pattern Sprint 0 used for real-Postgres tests (SAF-11's `SAF_TEST_POSTGRES_URL` gating). | Backend Engineer (Task 1.5/1.6 owners) |
| 15 | **Testing** | The clarification loop's non-termination case (Risk #9) has no automated test, only manual verification. | Medium | Medium | Add an explicit test with a deliberately bounded-adversarial input as part of Task 1.11's testing requirements — see [05-test-strategy.md](05-test-strategy.md). | Backend Engineer (Task 1.11 owner) |
| 16 | **Delivery** | CI has never executed on a real GitHub Actions runner (Task 1.23). | High (certain, until it happens) | Medium | Task 1.23 exists specifically to close this; gated only on push authorization, the user's call. | DevOps Engineer / User (authorization) |
| 17 | **Delivery** | The merged slice's larger scope (23 tasks vs. the original 20) takes meaningfully longer than one sprint, given the sprint carries an XL-sized single Vertical Slice with no smaller independently-shippable increment. | Medium | High | [03-dependency-map.md](03-dependency-map.md)'s parallel-work plan (six day-one starting points) is designed specifically to compress the schedule; if real progress shows the sprint genuinely won't finish, that's a Sprint Exit Gate conversation ([PROJECT_PLAYBOOK.md](../../../PROJECT_PLAYBOOK.md)), not a reason to cut a acceptance criterion silently. | Product Owner / Delivery Lead |
