# Sprint 1 Risk Register

**Revised 2026-07-17** for the corrected, PDR-synchronized task numbering ([02-vertical-slice-catalog.md](02-vertical-slice-catalog.md)). The two architecture-drift risks introduced by the 2026-07-16 revision's Digital Twin pull-forward and Project Type field (former #7, #8) are **removed** along with the scope that caused them — a risk register's job is to name real risk in real scope, not to accumulate entries from scope that's since been reverted.

| # | Category | Risk | Probability | Impact | Mitigation | Owner |
|---|---|---|---|---|---|---|
| 1 | **Business** | Discovery structures requirements the business stakeholder didn't actually mean, and it isn't caught before confirmation. | Low | High | The `Clarification` mechanism catches ambiguity before it's silently resolved; Project Charter confirmation (Task 1.15) is the human checkpoint; "Request Changes" always available. | Product Owner |
| 2 | **Business** | The real LLM call (Task 1.5) incurs unexpected cost at scale. | Medium | Medium | `CostBudget`/`UsageRecord` mechanism already designed in the LLM Gateway context (Sprint 0); confirm budget guardrails are active before any real traffic. | Backend Engineer (Task 1.5 owner) |
| 3 | **UX** | The Discovery experience reads as "a document generator with an AI assist" rather than "a consulting platform" if Quick Wins are skipped or rushed. | Medium | Medium | Quick Wins are explicit acceptance criteria on Tasks 1.6, 1.11, 1.15, 1.17 — not optional polish. | Frontend Engineer (Task 1.11/1.15/1.17 owners) |
| 4 | **UX** | Accessibility gaps (placeholder-as-label, unlabelled clarification inputs, unstructured confirmation content) ship if not explicitly verified. | Medium | Medium | Each fix is a named acceptance criterion on Tasks 1.9, 1.11, 1.15 — not a general "accessibility pass." | Frontend Engineer |
| 5 | **Architecture** | Real LLM integration (Task 1.5) reveals `LlmProviderPort`'s contract doesn't actually fit a real provider's behavior. | Low | High | **Stop, per [.ai/README.md](../../../.ai/README.md)'s "stop, explain impact, recommend an ADR" rule** — do not patch around a port mismatch silently. | Backend Engineer (Task 1.5 owner) |
| 6 | **Architecture** | `CapabilityResolverPort`'s first real adapter (Task 1.7) reveals the port's contract doesn't generalize past the single-provider case it was designed against. | Low | Medium | Same stop-and-recommend-ADR rule; the new contract-test suite (Task 1.7) is designed to surface this early. | Backend Engineer (Task 1.7 owner) |
| 7 | **Workflow** | The clarification loop (Tasks 1.10/1.11) doesn't terminate for a pathological or adversarial idea input. | Low | Medium | Bound the number of clarification rounds explicitly (application-layer safeguard); surface a clear "needs manual review" outcome past the bound. | Backend Engineer (Task 1.10 owner) |
| 8 | **Workflow** | `WorkflowRun` state gets corrupted by a concurrent clarification answer and a workflow-engine timeout racing each other. | Low | Medium | The in-memory workflow engine's existing state-transition guarantees, exercised for the first time by a real multi-round workflow — verify explicitly in Task 1.16's testing, not assumed to hold. | Backend/Workflow Engineer (Task 1.16 owner) |
| 9 | **Security** | Anthropic API credentials are mishandled during Task 1.5's real-call implementation. | Low | High | Mandatory: credentials only through `SecretsVaultPort`; PR review checklist ([PROJECT_PLAYBOOK.md](../../../PROJECT_PLAYBOOK.md)) checks this specifically. | Backend Engineer (Task 1.5 owner) + PR reviewer |
| 10 | **Security** | The new `CapabilityResolverPort` adapter (Task 1.7) resolves a capability without the same Zero Trust `RequestContext` discipline every other port already requires. | Low | Medium | `RequestContext` is a mandatory first parameter on every port method by existing convention — verified structurally at code review. | Backend Engineer (Task 1.7 owner) |
| 11 | **Documentation** | The two new persistence packages (Tasks 1.4, 1.13) ship without READMEs, failing `require-package-readmes.mjs`. | Low | Low | Already mechanically enforced by CI — cannot merge without it. | Owners of Tasks 1.4, 1.13 |
| 12 | **Testing** | Real-LLM-backed tests (Tasks 1.5, 1.6) are flaky or slow, discouraging developers from running them. | Medium | Medium | Follow the same env-gated real-provider pattern Sprint 0 used for real-Postgres tests (SAF-11's `SAF_TEST_POSTGRES_URL` gating). | Backend Engineer (Task 1.5/1.6 owners) |
| 13 | **Testing** | The clarification loop's non-termination case (Risk #7) has no automated test, only manual verification. | Medium | Medium | Add an explicit test with a deliberately bounded-adversarial input as part of Task 1.10's testing requirements — see [05-test-strategy.md](05-test-strategy.md). | Backend Engineer (Task 1.10 owner) |
| 14 | **Delivery** | CI has never executed on a real GitHub Actions runner (Task 1.19). | High (certain, until it happens) | Medium | Task 1.19 exists specifically to close this; gated only on push authorization, the user's call. | DevOps Engineer / User (authorization) |
| 15 | **Delivery** | The Anthropic real-call dependency (Task 1.5) blocks Task 1.6, which is on the critical path. | Medium | High | Start Task 1.5 on day one, in parallel with everything else (per [03-dependency-map.md](03-dependency-map.md)) precisely because it's both early-startable and critical-path. | Backend Engineer (Task 1.5 owner) |
| 16 | **Delivery** | Recurrence risk: a future re-interpretation of Sprint 1 scope (as happened twice already — once toward a form-based wizard, once toward a Digital Twin/Dashboard pull-forward) causes further execution-package churn without changing the underlying, already-approved Product Design Review. | Medium | Medium | Treat the Product Design Review as the single source of truth for *what* Sprint 1 builds; treat this execution package as the source of truth for *how* it's sequenced — any future proposal that changes screens, fields, or artifacts should be checked against the PDR first, not built into the execution package directly. | Product Owner / Delivery Lead |
