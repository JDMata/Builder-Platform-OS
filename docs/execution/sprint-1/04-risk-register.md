# Sprint 1 Risk Register

Consolidates risks already on record ([12-risks-and-technical-debt.md](../../architecture/12-risks-and-technical-debt.md), [Known Risks](../../governance/sprint-0-exit-gate/07-known-risks.md), the Product Design Review's UX/Future risk findings) with new risks specific to this execution plan. No risk here is invented for volume — each is a real, traceable consequence of a specific task in [02-vertical-slice-catalog.md](02-vertical-slice-catalog.md).

| # | Category | Risk | Probability | Impact | Mitigation | Owner |
|---|---|---|---|---|---|---|
| 1 | **Business** | Discovery structures requirements the business stakeholder didn't actually mean, and it isn't caught before approval. | Low | High | The `Clarification` mechanism exists exactly to catch ambiguity before it's silently resolved; the review/approve gate (VS-2) is the human checkpoint; "Request Changes" always available. | Product Owner |
| 2 | **Business** | The real LLM call (Task 1.5) incurs unexpected cost at scale. | Medium | Medium | `CostBudget`/`UsageRecord` mechanism already designed in the LLM Gateway context (Sprint 0); confirm budget guardrails are active before any real traffic, not just contract-tested. | Backend Engineer (Task 1.5 owner) |
| 3 | **UX** | The Discovery experience reads as "a document generator with an AI assist" rather than "a consulting platform" if Quick Wins are skipped or rushed. | Medium | Medium | Quick Wins are explicit engineering tasks (1.6, 1.11, 2.5, 2.7), not optional polish — tracked with the same acceptance criteria discipline as any other task. | Frontend Engineer (Task 1.11/2.5/2.7 owners) |
| 4 | **UX** | Accessibility gaps (placeholder-as-label, unlabelled clarification inputs, unstructured review content) ship if not explicitly verified. | Medium | Medium | Each fix is a named acceptance criterion on Tasks 1.9, 1.11, 2.5 — not a general "accessibility pass" that can be silently dropped under deadline pressure. | Frontend Engineer |
| 5 | **Architecture** | Real LLM integration (Task 1.5) reveals `LlmProviderPort`'s contract doesn't actually fit a real provider's behavior (e.g., streaming, rate-limit shape). | Low | High | If this happens: **stop, per [.ai/README.md](../../../.ai/README.md)'s "stop, explain impact, recommend an ADR" rule** — do not patch around a port mismatch silently. | Backend Engineer (Task 1.5 owner) |
| 6 | **Architecture** | `CapabilityResolverPort`'s first real adapter (Task 1.7) reveals the port's contract doesn't generalize past the single-provider case it was designed against. | Low | Medium | The same stop-and-recommend-ADR rule applies; the new shared contract-test suite (Task 1.7's acceptance criteria) is designed to surface this before it reaches production code. | Backend Engineer (Task 1.7 owner) |
| 7 | **Workflow** | The clarification loop (Task 1.10/1.12) doesn't terminate for a pathological or adversarial idea input. | Low | Medium | Bound the number of clarification rounds explicitly (an application-layer safeguard, not a port/contract change); surface a clear "needs manual review" outcome past the bound rather than looping forever. | Backend Engineer (Task 1.10 owner) |
| 8 | **Workflow** | `WorkflowRun` state gets corrupted by a concurrent clarification answer and a workflow-engine timeout racing each other. | Low | Medium | The in-memory workflow engine adapter's existing state-transition guarantees (proven in Sprint 0) are exercised for the first time by a real multi-round workflow — verify explicitly in Task 1.12's testing, not assumed to hold. | Backend/Workflow Engineer (Task 1.12 owner) |
| 9 | **Security** | Anthropic API credentials are mishandled (hardcoded, logged, or committed) during Task 1.5's real-call implementation. | Low | High | Mandatory: credentials only through `SecretsVaultPort`, verified as an explicit acceptance criterion, not an assumption; PR review checklist ([PROJECT_PLAYBOOK.md](../../../PROJECT_PLAYBOOK.md)) checks this specifically. | Backend Engineer (Task 1.5 owner) + PR reviewer |
| 10 | **Security** | The new `CapabilityResolverPort` adapter (Task 1.7) resolves a capability without the same Zero Trust `RequestContext` discipline every other port already requires. | Low | Medium | `RequestContext` is a mandatory first parameter on every port method by existing convention (ARCHITECTURE_PRINCIPLES.md) — verified structurally at code review, not a new check to invent. | Backend Engineer (Task 1.7 owner) |
| 11 | **Documentation** | The two new persistence packages (Tasks 1.4, 2.2) ship without READMEs, failing the existing `require-package-readmes.mjs` fitness function. | Low | Low | Already mechanically enforced by CI — cannot merge without it; listed here for completeness, not because it's a live risk. | Backend Engineer (Task 1.4/2.2 owners) |
| 12 | **Testing** | Real-LLM-backed tests (Task 1.5, 1.6) are flaky or slow, discouraging developers from running them. | Medium | Medium | Follow the same pattern Sprint 0 used for real-Postgres tests (SAF-11's `SAF_TEST_POSTGRES_URL` gating) — an equivalent env-gated real-provider suite, skipped cleanly by default, run deliberately and in CI. | Backend Engineer (Task 1.5/1.6 owners) |
| 13 | **Testing** | The clarification loop's non-termination case (Risk #7) has no automated test, only manual verification. | Medium | Medium | Add an explicit test with a deliberately bounded-adversarial input as part of Task 1.10's testing requirements — see [05-test-strategy.md](05-test-strategy.md). | Backend Engineer (Task 1.10 owner) |
| 14 | **Delivery** | CI has never executed on a real GitHub Actions runner (Task 1.13) — the largest verified-vs-proven gap carried from the Sprint 0 Exit Gate. | High (certain, until it happens) | Medium | Task 1.13 exists specifically to close this; gated only on push authorization, which remains the user's call, not a technical blocker. | DevOps Engineer / User (authorization) |
| 15 | **Delivery** | The Anthropic real-call dependency (Task 1.5) blocks Task 1.6, which is on the critical path — any delay here delays the whole sprint. | Medium | High | Start Task 1.5 on day one, in parallel with everything else (per [03-dependency-map.md](03-dependency-map.md)'s recommended order) precisely because it's both early-startable and critical-path. | Backend Engineer (Task 1.5 owner) |

## Risk categories with no live findings

**None of the fifteen risks above is a "manufactured" risk added to satisfy the category list** — but for completeness: no additional Governance-, Observability-, or Persistence-category risk beyond what's captured above was found; Sprint 0's existing risk register ([12-risks-and-technical-debt.md](../../architecture/12-risks-and-technical-debt.md)) already covers those categories at the platform level, and nothing in Sprint 1's scope introduces a new instance of them.
