# Sprint 1 Success Metrics

Defines how Sprint 1's success is measured at the Sprint Exit Gate ([PROJECT_PLAYBOOK.md](../../../PROJECT_PLAYBOOK.md)). Metrics are honest counts and pass/fail checks against this sprint's own real scope — not fabricated percentages against an idealized target no one committed to.

| Metric | Definition | Sprint 1 target |
|---|---|---|
| **Completed Vertical Slices** | Count of VS-1/VS-2 meeting their full Definition of Done ([02-vertical-slice-catalog.md](02-vertical-slice-catalog.md)). | 2 of 2. |
| **Capability Coverage** | Count of capabilities Sprint 1 introduces that are registered *and* resolved through a real `CapabilityResolverPort` adapter (not composed directly, as Sprint 0's single-provider demo did). | 1 of 1 (`structure-business-requirement`). |
| **Digital Twin Coverage** | Not measurable this sprint — `context-digital-twin` doesn't exist until Sprint 7. Tracked instead as **Interim Traceability Coverage**: the proportion of this sprint's generated artifacts (`RequirementDocument`, `Requirement`, `Clarification`, `AcceptanceCriterion`, `Project`) that are real, persisted, ID-referenceable rows with a recorded `AgentInvocation` where applicable. | 100% (5 of 5 artifact types) — the honest, currently-available substitute metric, per the Engineering Planning Principles' interim standard. |
| **Automated Test Coverage** | Every task in [02-vertical-slice-catalog.md](02-vertical-slice-catalog.md) has its test layer(s) from [05-test-strategy.md](05-test-strategy.md) passing in CI, plus the one required end-to-end demo (Task 2.6). | 20 of 20 tasks with a passing, owned test layer; 1 of 1 end-to-end demo passing unattended. |
| **Documentation Coverage** | Every new/changed package has a current README; every architecture doc a change makes inaccurate is updated in the same PR (Documentation First principle). | 2 new package READMEs (`persistence-postgres/requirements`, `persistence-postgres/project`); [02-domain-model.md](../../architecture/02-domain-model.md) updated to confirm `RequirementDocument`/`Clarification`/`AcceptanceCriterion`/`Project` are implemented. |
| **Telemetry Coverage** | Every task with a Telemetry requirement in [02-vertical-slice-catalog.md](02-vertical-slice-catalog.md) has real OTel spans/structured logs, verified against a live collector (the same bar SAF-16 set in Sprint 0), not just present in code. | Idea submission, each structuring invocation, each clarification answer, approval attempts (including rejections), Project creation, event emission — all traced. |
| **Workflow Coverage** | The Discovery `WorkflowDefinition` exercises every `WorkflowRunStatus` value the port defines (`pending`, `running`, `awaiting_approval`, `completed`) at least once in a real run; `failed`/`cancelled` are exercised in tests, not necessarily in the demo. | 4 of 4 "happy path" statuses in the demo; `failed`/`cancelled` covered by Task 1.12/2.4's automated tests. |
| **Business Value Delivered** | Not a count — a direct check against the Sprint 1 mission: *"a user transforms a business idea into an approved software project through an AI-guided discovery experience."* Pass/fail, verified by the demonstration plan ([06-demonstration-plan.md](06-demonstration-plan.md)), not a proxy metric. | Pass — both demos together prove the full mission statement end to end. |
| **Technical Debt Introduced** | Every new item logged, classified by severity, given a resolution timeline, per [TECHNICAL_DEBT_POLICY.md](../../../TECHNICAL_DEBT_POLICY.md) — tracked, not assumed to be zero. | Recorded honestly at the Sprint Exit Gate; anticipated candidates: any env-gated real-provider test skip pattern (Risk #12, [04-risk-register.md](04-risk-register.md)) carried forward the same way SAF-11's real-Postgres gating was. |
| **Technical Debt Removed** | Count of previously-accepted debt items closed this sprint. | 1 (`drizzle-orm` High-severity CVE, Task 1.2) + 1 (CI never run on a real runner, Task 1.13) — both carried forward from the Sprint 0 Exit Gate's own findings. |
| **Sprint Goal Completion** | Binary — did Sprint 1 achieve its stated exit criteria (a user submits a requirement through a real UI, it runs through a real workflow, and produces a real, traceable, tested, documented, telemetry-emitting, capability-registered result)? | Pass/fail, assessed at the Sprint 1 Exit Gate against this exact statement, not a softened restatement of it. |

## Why no invented percentages

Metrics like "Capability Coverage: 100%" or "Digital Twin Coverage: 100%" look precise but would be **meaningless precision** if computed against a denominator this sprint didn't actually commit to (e.g., "% of all conceivable future capabilities" has no defined denominator yet). Every metric above is scoped to what Sprint 1 actually built, per its own approved backlog — a metric with an undefined or inflated denominator is worse than no metric, because it invites false confidence.
