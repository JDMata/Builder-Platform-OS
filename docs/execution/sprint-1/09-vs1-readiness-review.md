### Sizing determination

**Is the slice too large?** No. Nineteen tasks sounds large in isolation, but the slice is Sprint 1's entire deliverable, not one of several — sized against that, it's appropriately scoped, and every task traces to a real, necessary piece of one coherent capability (see [03-dependency-map.md](03-dependency-map.md)'s critical path — nothing on it is incidental).

**Is the slice too small?** No — it's the smallest unit that still constitutes the complete, honest "idea to approved project" capability the Sprint 1 mission promises. A smaller cut (e.g., stopping at structured-but-unapproved requirements) would leave no durable business outcome, which the Product Design Review's own Executive Design Review already rejected as a valid stopping point for the mission statement.

**Missing dependencies?** One found and corrected — see Section 3. No others identified after checking: the `requirements-analyst` prompt file (exists), `testing-kit`'s contract-test pattern (extensible, already proven 8 times over in Sprint 0), and `PolicyEnginePort`'s real OPA adapter (exists, wired into `apps/api-gateway`, just not yet called by the new routes — now an explicit acceptance criterion).

**Mixing multiple business capabilities?** No. Every task serves exactly one capability — "idea becomes approved project." Login is the one task that could be argued as a separate, orthogonal concern (authentication is a platform capability, not a Discovery-specific one) — but at S effort, reusing an already-built backend flow, it doesn't rise to "mixing capabilities"; it's a one-screen prerequisite, the same category as any other enabling task in the slice.

**Introducing unnecessary technical complexity?** One real instance found and corrected — the `SecretsVaultPort` acceptance criterion implied a new adapter needed to be built, when the platform's own accepted technical debt baseline says the existing dev-only `.env` adapter remains sufficient until a second provider or secret rotation is needed (neither triggered by this slice). Corrected, not left in place — see Section 3. No other unnecessary complexity found: `CapabilityResolverPort`'s real adapter (Task 1.7) was deliberately, specifically justified (a real workflow needing dynamic resolution, not a demo script composing it directly) and is not revisited by this review, since nothing new changes that reasoning.

**Should the slice be split?** No. Splitting further would either produce a partial slice with no durable business outcome (violating "every Vertical Slice must deliver user-visible business value") or an artificial seam inside one continuous capability (which this review, the Product Design Review, and the prior Execution Package revision have all independently confirmed is the wrong shape). **The slice is confirmed as correctly sized and correctly bounded.**

## 3. Final Acceptance Criteria

Consolidated, corrected, and confirmed — supersedes any prior version implicit in [02-vertical-slice-catalog.md](02-vertical-slice-catalog.md) prior to this review's two edits:

1. A user reaches the idea-submission screen only after authenticating (Task 1.1); there is no dashboard step in between.
2. Submitting non-empty idea text, scoped to a `Workspace`, always produces a persisted draft `RequirementDocument` (Tasks 1.3, 1.4, 1.8) — **and the submitting user's permission to do so within that `Workspace` is checked via the real `PolicyEnginePort` (OPA) adapter, not inferred from authentication alone** *(corrected by this review)*.
3. Structuring runs against a real Anthropic API call (Task 1.5), with credentials read through `SecretsVaultPort`'s existing interface — **its current dev-only adapter is sufficient; no new adapter is required** *(corrected by this review)*.
4. The `requirements-analyst` capability (Task 1.6), resolved dynamically via `CapabilityResolverPort`'s first real adapter (Task 1.7), produces at least one `Requirement` and a suggested `Project` name from realistic input, and raises a `Clarification` rather than guessing wherever confidence is low.
5. Every `Clarification` is answerable (Task 1.10) and re-triggers exactly one structuring pass, quoting its triggering idea fragment and showing staged progress (Task 1.11) — never re-asking an already-answered question.
6. Confirmation (the Project Charter screen, Task 1.15) is blocked while any `Clarification` is outstanding, shows every requirement with a derived confidence badge, and is screen-reader-navigable.
7. Confirming (Task 1.14) creates exactly one `Project` (using the agent-derived name) and emits `requirements.document.captured.v1`, verifiably in the outbox — **and the confirming user's permission to approve is checked via the same `PolicyEnginePort` discipline as item 2** *(corrected by this review)*.
8. The full workflow run (Task 1.16) transitions `pending → running → awaiting_approval → running → ... → awaiting_approval → completed`, emitting every expected event at each transition.
9. Project Ready (Task 1.17) shows the real created `Project` and a static "what happens next" panel that previews, without implying the existence of, future capabilities.
10. `pnpm run demo:sprint1` (Task 1.18) runs unattended end to end, against real adapters at every layer, and produces inspectable evidence (a real `Project` row, a real outbox event).
11. CI executes successfully on a real GitHub Actions runner for the first time (Task 1.19), once push is authorized.
