# 8. Sprint 0 Lessons Learned

## Patterns worth repeating in Sprint 1

1. **"Extract on second occurrence, not first."** Applied consistently — `resilience-kit` (SAF-10, once `mcp-core` needed what `llm-core` had), `createPostgresOutboxEventBus()` (SAF-6, once `worker` needed what `orchestrator` had), `test-support.ts`/`test-fake-dependencies.ts`'s shared shape. Never once did this codebase extract an abstraction speculatively ahead of a real second consumer, and never once did a real second consumer get copy-pasted instead of factored. Keep doing this.
2. **Verify against real infrastructure, not mocks, whenever the thing being tested is the infrastructure integration itself.** Every real bug found this sprint — the two-Postgres-role/RLS-bypass discovery (SAF-14), the sequence-grant gap (SAF-6), the cross-package migration-tracking collision (SAF-15), the `docker compose ps --format json` array-vs-object shape mistake (SAF-15), the workspace-symlink-resolves-to-dist-not-src gap (SAF-19) — was found by running the real thing, never by reasoning about it in the abstract. Contract tests against fakes are necessary but not sufficient; this sprint's actual bug-finding record came from hitting real Postgres, real Keycloak, real OPA, real Docker, repeatedly, including deliberately from a clean/reset state.
3. **When a fitness function or CI check is added, prove it fails before trusting that it passes.** Every new dependency-cruiser rule and every new `tools/scripts/*.mjs` check this sprint was verified by injecting a real violation and confirming it was caught, then reverting — not just run once and trusted because it exited 0. A check that has never been seen to fail is unverified, regardless of how many times it's passed.
4. **Document the gap the moment it's created, not when someone rediscovers it.** The `CapabilityProvider`/`CapabilityBinding` naming collision, the `plugin-import-boundary` rule's narrower-than-aspirational scope, the `Notification` context's ambiguity — all were written down explicitly at the moment they were found, with a stated trigger for when they'd need revisiting. This is why this exit-gate audit found so few *new* architectural surprises: most of what it confirms was already self-reported.

## Mistakes worth not repeating

1. **A written-ahead document (`CONTRIBUTING.md`) drifted from reality and nothing caught it for the rest of the sprint.** It was written early, describing an *intended* infra shape, and never revisited once SAF-13 actually decided something different (no Redis/MinIO, a different compose file layout). No story's own "update the docs this story touches" discipline had a reason to open a file about environment setup. **Lesson:** a periodic "does onboarding actually still work" check (exactly what Phase 3 of this exit gate did) needs to happen more than once per sprint for a fast-moving Sprint 0, not only at the very end.
2. **Backlog bookkeeping lagged actual completion twice** (SAF-23 was functionally done at SAF-14, never marked; SAF-18 was partially done very early, never fully closed). Neither caused a real defect, but both meant the backlog file — this project's single source of truth for "what's done" — was quietly wrong for most of the sprint. **Lesson:** when a story's work incidentally satisfies a *different* backlog item, mark that item done in the same PR, not "eventually."
3. **A first attempt at a new mechanism was sometimes subtly wrong in a way only real execution surfaced** — the `plugin-import-boundary` rule's first draft matched only `src/`, missing that real imports resolve through `dist/`; the migration-tracking fix's first attempt (`migrationsSchema`) collided with the migration files' own `CREATE SCHEMA` statements. Neither is a process failure — both were caught before merge, by the "verify by injecting a real violation" discipline above — but both are a reminder that a plausible-sounding first design for a new dependency-cruiser rule or a new infra interaction should be treated as a hypothesis, not a fact, until proven against the real repo.
4. **Assuming pnpm's workspace resolution behaves like a flat `node_modules`.** Twice this sprint (the plugin-import-boundary rule, and `tools/sprint0-demo`'s original design as a loose script) a design was built on an implicit assumption about how imports resolve, and both times the assumption was wrong in a way that only surfaced by actually running the thing. **Lesson:** for anything touching cross-package resolution in this specific pnpm-workspace setup, test the actual resolved path/behavior before designing around an assumption about it, even when the assumption seems obviously correct.

## What made Sprint 0 successful despite its length

Ten explicit "review pauses" happened over the course of the sprint (the mid-sprint architecture inventory review, the principal-architect self-review, this exit gate) rather than one long uninterrupted build. Each caught something real (renamed adapters left unrenamed, the Notification context's fate, this exit gate's documentation-drift and bookkeeping findings) that a single continuous push to "finish the backlog" would very plausibly have missed or deferred indefinitely. The lesson isn't "review more" in the abstract — it's that a genuinely independent, skeptical pass (assuming something is wrong until proven otherwise, exactly this document's own framing) finds different things than the builder's own steady, story-by-story diligence does, even when that diligence is high. Both are needed; neither substitutes for the other.
