## What's missing, and the smallest-fit recommendation (not implemented now)

**Assumptions, Constraints, and Dependencies have no home in the current domain model.** The smallest-fit correction, for a future sprint to actually implement (not this review): extend `RequirementKind`'s existing union (`"business" | "functional" | "non-functional" | "user-story"`) to also include `"assumption" | "constraint" | "dependency"` — reusing the already-designed `Requirement` shape rather than introducing three new aggregates. This mirrors exactly how `ADR-0021` already folded "a User Story is a `kind` of Requirement, not a separate aggregate" — the same reasoning extends cleanly to these three. **Not implemented as part of this review** — it touches `packages/context-requirements`'s domain code, which is Sprint 1 implementation work, explicitly out of scope for a Product Design Review. Recorded here as a recommendation for Sprint 1's own implementation phase or Sprint 2's planning, whichever comes first.

## General principle validated across every artifact above

**Nothing generated by an AI capability is directly, freely editable as raw text** — the pattern Sprint 1 already established (structure through the agent + clarification loop, never a text box) is confirmed as the right pattern for every future artifact type too (Functional Specs, Architecture Summaries, etc.), not something to relax "for convenience" once more artifact types exist. **Everything that constitutes an approval boundary (a `RequirementDocument` reaching `approved` status) freezes what it approved** — future artifacts should adopt the same freeze-on-approval discipline rather than inventing a new one each time a new artifact type is added.

## What should be generated automatically vs. require approval — summary rule

- **Automatic:** extraction, structuring, gap/clarification detection — anything the agent can produce as a *proposal*.
- **Requires approval:** anything that changes system-of-record state (an approved document, a created `Project`) or feeds a downstream capability that assumes correctness (a future Functional Specification generated from this Project's approved requirements must be able to trust they were actually approved, not merely drafted).

This rule is already exactly what SAF-50 implements for Sprint 1's own scope — validated here as the correct rule to carry forward, not something Sprint 1 got wrong.
